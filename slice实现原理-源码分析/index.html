数组没那么常用，切片更常用，切片的长度不是固定的，可以向切片中追加元素，它会在余量不足时自动扩容。

cmd\compile\internal\types\type.go 中的 NewSlice 函数是在编译期间用于创建切片类型

NewSlice 返回的结构体中的 Extra 字段是下面的 Slice 结构体，它只包含切片内元素类型

即，切片内元素的类型在编译期间就确定了，编译器确定了类型后，会将类型存储在 Extra 字段中，帮助程序在运行时动态获取。

上面这个是编译期间切片类型，但是在运行时，切片可以由如下的结构体表示：

Data 是指向数组的指针
Len 是当前切片的长度
Cap 是当前切片的容量，即 Data 数组的大小

Data 是一片连续的内存空间，可以存储切片中的全部元素，数组中的元素只是逻辑上的概念，底层存储都是连续的，可以理解成切片是一片连续的内存空间加上长度和容量的标识

切片和数组密切相关，切片引入了一个抽象层，提供了对数组中部分连续片段的引用，我们可以在运行时修改它的长度和范围，当切片底层的数组长度不足时，会触发扩容，切片指向的数组可能会变化，不过在上层看来，切片没有发生变化，上层只需要和切片打交道，不关心底层数组的变化

编译器在编译期间获取数组大小，读取数组中的元素，因为数组的内存固定且连续，多数操作都会直接读写内存的特定位置，但是切片是运行时才会确定的结构，所有操作还需依赖 go 语言的运行时，下面会结合运行时介绍切片常见的操作背后的原理。

三种初始化切片的方式

  1. 通过下标 获得数组或切片的一部分
  2. 字面量 初始化新的切片
  3. make 创建切片

使用下标创建切片最原始也最接近汇编语言，编译器会将 arr [0:3] 或者 slice [0:3] 等语句转换成 OpSliceMake 操作，我们验证一下：

SliceMake 操作会接收 4 个参数而创建新的切片：
元素类型 数组指针 切片大小 切片容量

这也是切片的几个字段，使用下标初始化切片不会拷贝原数组或原切片中的数据，而是创建一个指向原数组的切片结构体，所以修改新切片的数据也会修改原切片。

步骤：

  1. 根据切片中的元素数量，对底层数组的大小进行推断，并创建一个数组
  2. 将这些字面量元素存储到初始化的数组中
  3. 创建一个同样指向 [3] int 类型的数组指针
  4. 将静态存储区的数组 vstat 赋值给 vauto 指针所指向的地址
  5. 通过 [:] 操作获取一个底层使用 vauto 的切片

第五步的 [:] 操作就是在使用下标创建切片，可见 [:] 操作是创建切片的最底层的方法

关键字 make 创建切片
如果使用字面量的方式创建切片，大部分工作会在编译期间完成，但当使用 make 创建切片时，很多工作都需要运行时的参与，调用方必须向 make 传递切片的大小和容量（可选）

类型检查期间的 typecheck1 函数会校验入参
src\cmd\compile\internal\gc\typecheck.go

不仅会检查 len 是否传入，还会保证传入的容量 cap 一定大于或等于 len
除了参数校验外，当前函数会将 OMAKE 节点转成 OMAKESLICE，中间代码生成的 cmd/compile/internal/gc.walkexpr 函数会依据下面两个条件转换 OMAKESLICE 类型的节点：

  1. 切片的大小和容量是否足够小
  2. 切片是否发送了逃逸，最终在堆上初始化

即，初始化数组并通过下标 [:3] 得到数组对应的切片，这两部分的操作都会在编译阶段完成，编译器会在栈上或静态存储区创建数组，并将 [:3] 转换成 OpSliceMake 操作

分析了主要由编译器处理的分支后，回到用于创建切片的运行时函数 makeslice，它主要是计算切片占用的内存空间，并申请在堆上的一片连续的内存

$ 内存空间 = 切片中元素的大小 × 切片容量 $

虽然编译期间可以检查出很多错误，但在创建切片的过程如果发生了以下错误，会直接触发运行时错误并 panic

  1. 内存空间的大小发生溢出
  2. 申请的内存大于最大可分配的内存
  3. 传入的长度小于 0，或长度大于容量

mallocgc 是用于申请内存的函数，这个函数的实现比较复杂，如果遇到了比较小的对象会直接初始化在 go 语言的调度器中的 P 结构里，而大于 32KB 的对象会在堆上初始化（需要了解 Go 的内存分配器）

老版本的 go 中，数组指针，长度和容量会合成到一个 slice 结构体，但是从 https://github.com/golang/go/commit/020a18c545bf49ffc087ca93cd238195d8dcc411#diff-d9238ca551e72b3a80da9e0da10586a4
之后，构建结构体 reflect.SliceHeader 的工作就交给了 runtime.makeslice 的调用方，该函数只会返回指向底层数组的指针，调用方会在编译期间构建切片的结构体

OSLICEHEADER 操作会创建 reflect.SliceHeader 结构体，其中包含数组指针，切片长度和容量，它是切片在运行时的表示：

因为大多数对切片类型的操作不需要直接操作原来的 slice 结构体，所以 SliceHeader 的引入能减少切片初始化的少量开销。

使用 len 和 cap 获取长度和容量，编译器会将它们看作两种特殊的操作，即 OLEN 和 OCAP

是否覆盖原变量的逻辑其实差不多，最大区别在于得到的新切片是否会赋值回原变量，如果我们选择覆盖原变量，就不需要担心切片发生拷贝而影响性能，因为 go 语言编译器已经对这种常见情况作了优化。

到这里我们已经清楚了 go 语言如何在切片容量足够时，向切片中追加元素，不过仍然需要研究切片容量不足时的处理流程

当切片容量不足时，会调用 runtime/slice.go growslice 函数为切片扩容，即为切片分配新的内存空间并拷贝原切片中的元素，先看新切片的容量是如何确定的

func growslice(et *_type, old slice, cap int) slice {
newcap := old.cap
doublecap := newcap + newcap
if cap > doublecap {
newcap = cap
} else {
if old.len < 1024 {
newcap = doublecap
} else {
for 0 < newcap && newcap < cap {
newcap += newcap / 4
}
if newcap <= 0 {
newcap = cap
}
}
}
Go
在分配内存空间之前，需要先确定新切片的容量，运行时根据切片的当前容量，选择不同的策略进行扩容：

  1. 如果期望容量大于当前容量的 2 倍，使用期望容量
  2. 否则
    1. 如果当前切片的长度小于 1024，将容量翻倍
    2. 否则，每次增加 1/4，直到新容量大于期望容量

这些代码只会确定切片的大致容量，还需要根据切片中的元素大小对齐内存，当数组中元素所占的字节大小为 1、8 或者 2 的倍数时，运行时会使用下面的代码对齐内存：

var overflow bool
var lenmem, newlenmem, capmem uintptr
switch {
case et.size == 1:
    lenmem = uintptr(old.len)
    newlenmem = uintptr(cap)
    capmem = roundupsize(uintptr(newcap))
    overflow = uintptr(newcap) > maxAlloc
    newcap = int(capmem)
case et.size == sys.PtrSize:
    lenmem = uintptr(old.len) * sys.PtrSize
    newlenmem = uintptr(cap) * sys.PtrSize
    capmem = roundupsize(uintptr(newcap) * sys.PtrSize)
    overflow = uintptr(newcap) > maxAlloc/sys.PtrSize
    newcap = int(capmem / sys.PtrSize)
case isPowerOfTwo(et.size):
    ...
default:
    ...
}

Go
runtime.roundupsize 函数会将待申请的内存向上取整，取整时会使用 runtime.class_to_size 数组，使用该数组中的整数可以提高内存的分配效率并减少碎片，我们会在内存分配一节详细介绍该数组的作用：

var class_to_size = [_NumSizeClasses]uint16{
0,
8,
16,
32,
48,
64,
80,
…,
}
Go
默认情况下，我们会将目标容量和元素大小相乘，得到占用的内存。
如果计算新容量时发生了内存溢出，或者请求内存超过上限，就会直接崩溃退出程序，这里不作过多了解。

var newlenmem, capmem uintptr
switch {
...

    capmem, _ = math.MulUintptr(et.size, uintptr(newcap))

Go
如果切片中元素不是指针类型，则调用 runtime.memclrNoHeapPointers 将超出切片当前长度的位置清空并在最后使用 runtime.memmove 将原数组内存中的内容拷贝到新申请的内存中。这两个方法都是用目标机器上的汇编指令实现的，这里就不展开介绍了。

runtime.growslice 函数最终会返回一个新的切片，其中包含了新的数组指针、大小和容量，这个返回的三元组最终会覆盖原切片。

var arr []int64
arr = append(arr, 1, 2, 3, 4, 5)

总结一下扩容的过程：当我们执行上述代码时，会触发 runtime.growslice 函数扩容，并传入期望的新容量 5，期望分配的内存大小为 40 字节，但由于切片中的元素大小等于 sys.PtrSize，所以运行时会调用 runtime.roundupsize 向上取整内存的大小到 48 字节，所以新切片的容量为 48 / 8 = 6。

3.2.5 拷贝切片 #
copy (a, b)，编译期间的 cmd/compile/internal/gc.copyany 也会分两种情况进行处理拷贝操作，如果当前 copy 不是在运行时调用的，copy (a, b) 会被直接转换成下面的代码：

n := len(a)
if n > len(b) {
n = len(b)
}
if a.ptr != b.ptr {
memmove(a.ptr, b.ptr, n*sizeof(elem(a)))
}
Go
上述代码中的 runtime.memmove 会负责拷贝内存。而如果拷贝是在运行时发生的，例如：go copy (a, b)，编译器会使用 runtime.slicecopy 替换运行期间调用的 copy，该函数的实现很简单：

func slicecopy(to, fm slice, width uintptr) int {
if fm.len == 0 || to.len == 0 {
return 0
}
n := fm.len
if to.len < n {
n = to.len
}
if width == 0 {
return n
}
…

size := uintptr(n) * width
if size == 1 {*(*byte)(to.array) = *(*byte)(fm.array)
} else {memmove(to.array, fm.array, size)
}
return n

}
Go
无论是编译期间拷贝还是运行时拷贝，两种拷贝方式都会通过 runtime.memmove 将整块内存的内容拷贝到目标的内存区域中：

golang-slice-copy

图 3-5 Go 语言切片的拷贝

相比于依次拷贝元素，runtime.memmove 能够提供更好的性能。需要注意的是，整块拷贝内存仍然会占用非常多的资源，在大切片上执行拷贝操作时一定要注意对性能的影响。

3.2.6 小结 #
切片的很多功能都是由运行时实现的，无论是初始化切片，还是对切片进行追加或扩容都需要运行时的支持，需要注意的是在遇到大切片扩容或者复制时可能会发生大规模的内存拷贝，一定要减少类似操作避免影响程序的性能。
