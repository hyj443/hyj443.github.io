Cache 函数做了什么事

全局的 cache map，保存了所有 CacheTable 的指针。而 CacheTable 的 name 字段保存它在 cache 中的 key 名，items 字段，是一个 map，键可以是任意类型，值是 CacheItem 指针类型。

做二次检查 table 是否存在，是为了并发安全，因为在别的协程上如果正好 Cache 调用，创建了 Table，能获取到了，就不用再创建了。

可见。Add 方法中，在调用 addInternal 方法之前，加了读写锁，但并没有在 addInternal 方法执行结束释放，而是在 addInternal 中，对相应资源读写结束之后释放，减少了持有锁的时间，精细了锁的粒度，提高了并行度。

addInternal 函数为什么要调用 expirationCheck，这是因为新加入的 item 的过期时间，可能比下次要进行删除的时间来得早，如果不更新下一次检查的时间，就导致这个 item 不能及时删除。

我们看看 expirationCheck：

expirationCheck 会先停止定时器，因为可能在别的地方调用此方法，而不是定时器自动调用的，为防止触发错误，故先停止定时器，然后遍历 item，通过判断当前时间 - 上次访问的时间是否 >= 生命周期，决定是否要删除数据，记录下 smallestDuration，即距离现在最近的，要删除的 item 的时间间隔，最后将设置定时器，在 smallestDuration 后再次触发 expirationCheck 进行数据更新。

table 的过期检查是利用了 Timer 定时器，但并没有在创建 CacheTable 时就初始化 Timer 相关的字段，而是在 Add 调用（加入了 item）时去初始化的。

因为 table 初始化后还没有数据，开启了 Timer 去定期执行也没有意义。

而且，每加入一个 item 就触发一次过期检查，这样更能及时地调整进行过期检查的时间点，并没有为每一个 item 都开一个定时器，过于消耗性能。只用了一个定时器，在 expirationCheck 方法中，遍历一遍检查 item 是否有过期，并找出最接近过期的 item，将它剩下的生命时间作为下一次过期检查的时间间隔。即确定下次检查的时间。

总结

太多的锁会造成性能损失，所以在读完数据或修改完数据后，要及时解锁，不然会其他协程会阻塞。

通过更新最小触发时间，来定时触发更新，用较小的资源维护大量数据。

同时注重了可扩展性，item 的 key 和 data 都可以是任意类型，用户还可以配置回调方法，方便维护数据，还可以自定义 logger。
