Go 并发的原则

Go 并发的哲学是：不通过共享内存来进行协程之间的通信，而是通过通信来实现共享内存。

先说前者，比如有一块共享的数据，协程a和b都要读写它，可通过加锁，二者来抢占。但这不可控，无法知道某一时刻轮到谁了，而且其他协程也能访问到它，存在一定的安全问题，如果只想通过这块共享的内存区域进行信息交换，还得限制不让不该访问的协程也来访问。

以上是共享内存的模型，常见实现就是共享变量和互斥锁，不同线程之间可以修改和读取某个共享变量，缺点是：内存的可见性是由开发者自己控制的。

而Go采用的是CSP模型，全称Communicating Sequential Processes，它采用了线程安全的队列进行通信，开发者不用考虑太多同步的问题，更加贴近业务去思考问题，不用考虑太多繁琐的底层问题。

CSP模型避免了线程之间出现竞争，毕竟诡异的线程竞争需要花很多时间debug。

go的管道其实就是一个队列，本文会仔细阐述，一个goroutine往管道里发送数据，另一个goroutine从管道中取出数据，go开发者无需关心管道如何保存和传递数据，只知道chan是goroutine之间的通信管道。

channel 的数据结构

源码版本为 go 1.15.6

看几个重点的字段：

buf 是指向管道的循环数组（环形的缓冲队列）的指针。
sendx，表示缓冲队列中当前发送元素的位置索引
recvx，表示缓冲队列中当前接收元素的位置索引
recvq，表示等待接收管道中的数据的goroutine的队列，即，保存了阻塞中的接收方
sendq，表示等待发送数据给管道的goroutine的队列，即，保存了阻塞中的发送方

下面这是sendq和recvq队列的实现，是一个双向链表，节点是sudog类型，是对goroutine的封装。
这些goroutine由于试图读取管道，或向管道发送数据，而被阻塞。

创建channel

为了方便后面统一用「管道」这词，我们知道make关键字用来创建管道。

通过汇编分析出，最终创建管道的函数是 makechan，它返回了hchan结构体的指针，所以我们可以在函数之间传递管道的指针，而不是管道本身。

makechan接收的t类型指针，包含管道中元素的类型信息，接收size，代表缓冲的容量。

注释写得很清楚，总结来说：先根据size和元素类型，计算出缓冲区所需分配的内存容量。

如果没有缓冲区，只需申请hchan结构体本身的一块内存即可。

如果缓冲区的元素不是指针类型，则给hchan和buf数组分配一段连续的内存。

如果缓冲区的元素是指针类型，则hchan和buf分开分配内存，GC会扫描管道中的元素。

然后初始化hchan的各个属性，比如元素的大小，元素的类型，缓冲区的容量，最后返回创建好的管道的指针。

向管道发送数据

使用 ch <- x 语句

chansend1 是 c <- x 经过编译后的入口函数。

我们直接看 chansend 就好，看看参数：
c是你要往里面发送数据的管道，ep是待发送的数据所在的内存地址，block表示发送操作是否阻塞式
最后返回的布尔值代表是否发送成功

函数中用的gopark函数，做的事是：将当前的goroutine置于等待状态（挂起），它让出处理器的使用权，等待被恢复（执行第一个参数unlockf），如果它传了 nil，那就是永久阻塞了，不会再恢复这个goroutine。

可见，向管道发送数据，存在几种情况：

  • 如果管道为nil
    • 如果发送是阻塞式的，则会一直阻塞这个发送的goroutine，同时抛错
    • 如果发送是非阻塞式的，阻塞不了，就直接不能发送，返回false
  • 如果管道不为nil，且管道是非阻塞的，且管道没有缓冲队列且recvq队列为空，或，管道有缓冲队列但已经满了
    • 即没有缓冲区的空间接收数据，也没有接收方接收数据，又不能阻塞，则不能发送，返回false
  • 如果管道已经关闭，是无法向已关闭的管道发送数据的，直接抛错
  • 如果recvq队列非空，说明有接收方在等待，说明此时缓冲队列是空的，直接绕过缓冲队列，直接将发送的数据拷贝给等待中的接收方
    • 即优先检查recvq中是否有等待“读”的goroutine
    • 从recvq队列中取出最早处于等待的sudog，直接将数据拷贝给它
  • 如果recvq队列为空，且缓存队列未满，则将数据拷贝到缓存队列的sendx索引上
  • 如果recvq队列为空，即没有接收方等待接收，且没有缓冲区或缓冲区已满
    • 如果管道是非阻塞式的，直接返回false，无法发送
    • 如果管道是阻塞式的，那发送操作就阻塞了
      • 获取当前g的指针，创建一个sudog，再挂载这次阻塞发送的有关信息，比如发送的管道，是否在select控制体中，待发送的数据的地址等
      • 然后将该sudog加入到当前管道的sendq队列中，作为阻塞中的发送方
      • 调用gopark方法，将当前的g阻塞，直到被唤醒
      • g被唤醒后会执行一些收尾工作，将一些属性重置，并释放sudog

我们仔细看看直接发送的逻辑，直接把数据发送给等待接收的goroutine

执行的是send函数：

如果接收方sg的存放数据元素的地址存在，则直接将ep地址上的数据，拷贝sg.elem

谈谈goready，它就是唤醒传入的接收方的goroutine，将它转成Grunnable可运行状态，会执行runqput函数将当前runq队列的末尾，等待被调度，这个runq队列存放可运行的goroutine

小结一下常见的几种情况：

  • 如果当前管道的recvq队列存在阻塞中的接收方，则直接将数据发送给等待中的接收方，将其设置为下一个运行的goroutine
  • 如果管道的recvq队列为空，且存在缓冲区，且还有剩余空间，则直接将数据存到缓冲区的sendx所在的位置上
  • 如果不满足上面情况，则会创建一个sudog，将这次阻塞发送的相关信息挂载到上面，将sudog加入到sendq队列中，当前g陷入了等待，等待其他g从管道中接收数据

向管道发送数据，存在几个会触发g调度的时机：

  • 发送数据时，如果管道的 recvq 队列中存在等待接收数据的 goroutine，立刻设置处理器的 runnext 属性，但不会立刻触发调度
  • 发送数据时，没有找到接收方，并且缓冲区满了，这时会将自己（goroutine）包裹成 sudog，加入到管道的 sendq 等待队列中，并调用 gopark 触发 Goroutine 的调度，让出处理器的使用权。

从管道接收数据

从管道接收数据有两种方式：
i := <- ch
i, ok := <- ch

经过编译，都会变成 ORECV 类型的节点，但后者会在类型检查阶段被转换成 OAS2RECV 类型。

分别被转换成 chanrecv1 和 chanrecv2 两种函数调用，但是都会实际调用 chanrecv：

chanrecv 函数做的事是：在管道c上接收数据，并将接收到的数据写入 ep 指向的内存。

函数的参数：c是管道，ep是元素的指针，block是否阻塞。selected为真表示被选择，received表示已接收到值

从管道接收数据的处理过程：

  • 如果管道为nil
    • 如果接收是非阻塞的，接收失败，返回两个false
    • 如果接收是阻塞的，则阻塞这个接收方，而且永久阻塞了
  • 如果管道不为nil，且管道非阻塞，且管道没有缓冲区且sendq队列没有发送方，或管道有缓冲区但缓冲区没有数据
    • 如果管道没有关闭，无法从管道接收数据，返回两个false
    • 如果管道关闭了，且此时没有可接收的数据，而ep可以接收到一个零值，返回true和false
  • 如果管道已经关闭，且管道的缓冲队列中没有数据
    • ep可以接收到一个零值，返回true和false
  • 如果sendq队列非空，获取一个阻塞中的发送方
    • 如果没有缓冲区，则直接将发送方的数据拷贝到ep
    • 否则将缓冲区中recvx索引上的元素拷贝给ep，再将sendq队列的队首的sudog的elem上的数据拷贝到缓冲区的recvx索引上，填补过来
  • 如果sendq队列为空，缓冲队列不为空，直接从缓冲队列中拷贝数据给ep
    • 获取sendx位置上的指针，如果ep存在，将它拷贝给ep
    • 将recvx上的内存清理，然后更新recvx索引，如果正好越界，重新归零，即环形队列
    • 缓冲队列的元素个数-1
  • 如果sendq队列为空，没有发送方，且缓冲队列为空
    • 如果是非阻塞状态，就无法接收数据，返回两个false
  • 如果sendq队列为空，并且是阻塞的，那么这次接收就是阻塞的
    • 获取当前的g，创建一个sudog，将接收数据的ep挂载到上面，将sudog加入到recvq队列
    • 调用gopark函数，将当前g阻塞，直到被恢复
    • 被唤醒后会进行一些清理工作，释放sudog，返回true !closed

仔细看看直接接收
当管道的sendq队列中有处于等待的goroutine时，会取出队首，调用recv函数。

可见

  • 如果管道没有缓冲区，直接调用recvDirect函数将 sendq 发送队列中的 sudog 存储的 elem 数据直接拷贝到目标内存地址 ep 中
  • 如果管道有缓冲区，将sendq发送队列中的数据拷贝到缓冲区中，释放了一个阻塞的发送方，发送队列的队头sudog的elem元素，会替换索引 recvx 所在位置的元素，recvx 上原有的元素会被拷贝到接收数据的变量的内存空间中

无论是那种情况，都会调用goready函数将当前处理器的runnext设置接收方的内存地址

小结一下 从管道接收数据的过程

  1. 如果管道为nil，直接调用 gopark 挂起当前的goroutine
  2. 如果管道已经关闭，并且缓冲区没有任何数据，直接返回
  3. 如果管道的 sendq 队列中存在等待的 goroutine，这说明要么不存在缓存队列，要么缓存队列满了，前者直接将等待中的sudog中的元素拷贝到接收变量所在的内存空间，后者将数据队列中recvx索引上的元素拷贝到目标变量，然后将等待队列sendq中的队首的元素，拷贝到buf队列的recvq索引上、
  4. 如果管道的缓冲区有数据，但未满，说明 sendq等待队列是空的，直接读取recvq索引上的对应的数据，拷贝到 ep，
  5. 。。。

总结一下从管道接收数据时，触发goroutine调度的两个时机
当管道为nil时
当缓冲区没有数据，并且也不存在数据的发送者时

关闭管道

当你不再使用一个管道（不再往里面发送值）时，别忘了把它关闭。

这就是告知接收方的那个 goroutine，发送工作已经完成。

编译器会将 close(ch) close 这个关键字转成 OCLOSE 节点，接着转成 closechan 函数执行。

所以，管道关闭的核心逻辑

  • 释放所有的接收者，将它们加入gList队列
  • 释放所有的发送者，将它们加入gList队列
  • 将glist中的元素都调用goready触发调度

关闭管道后，对于处于等待的接收方来说，会收到一个相应类型的零值
对于等待的发送方来说，会直接panic

close的逻辑是先锁住管道，然后把所有的接收方和发送方连成一个sudog链表，再解锁
然后再将所有sudog逐个全部唤醒
被唤醒的g，继续执行它的逻辑
接收方会进行一些扫尾工作，返回，selected返回true，received则要根据管道是否关闭，返回不同的值，在这里管道已经关闭，则返回false
发送方也会继续执行chansend 函数里 gopark 函数之后的代码，但检测到管道已经关闭，会panic
