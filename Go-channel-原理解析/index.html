Go 并发的原则

Go 并发的哲学是：不通过共享内存来进行协程之间的通信，而是通过通信来实现共享内存。

先说前者，比如有一块共享的数据，协程 a 和 b 都要读写它，可通过加锁，二者来抢占。但这不可控，无法知道某一时刻轮到谁了，而且其他协程也能访问到它，存在一定的安全问题，如果只想利用这块共享的内存区域进行信息交换，还得限制同一时间能读写这个变量的线程数量。

以上是共享内存的模型，常见实现是使用共享内存加互斥锁，Go 也提供了这种方案，但缺点是：内存的可见性是由开发者自己控制的。

Go 提供了 CSP 模型，全称 Communicating Sequential Processes 通信顺序进程，它采用了线程安全的队列进行通信，开发者不用考虑太多同步的问题，更加贴近业务，不用考虑太多繁琐的底层问题。

CSP 模型避免了线程之间出现竞争，诡异的线程竞争 debug 起来很花时间。

go 的管道其实就是一个队列，本文会仔细阐述，一个 goroutine 往管道里发送数据，另一个 goroutine 从管道中取出数据，二者能独立运行，不存在直接的关联，go 开发者无需关心管道如何保存和传递数据，但通过了管道间接完成通信。

channel 的数据结构

源码版本为 go 1.15.6

看几个重点的字段：

buf，是指向一个循环缓冲队列的指针。
sendx，表示缓冲队列中当前发送元素的位置索引
recvx，表示缓冲队列中当前接收元素的位置索引
recvq，表示等待接收管道中的数据的 goroutine 的队列，即，保存了阻塞中的接收方
sendq，表示等待发送数据给管道的 goroutine 的队列，即，保存了阻塞中的发送方

下面是 sendq 和 recvq 队列的实现，是一个双向链表，节点是 sudog 类型，是对处于等待的 goroutine 的封装，sudog 有前驱指针和后继指针，构成双向链表。

创建 channel

后面统一用「管道」这词，我们用 make 关键字用来创建管道。

通过 汇编 分析出，最终创建管道的函数是 makechan，它返回了 hchan 结构体的指针，所以我们可以在函数之间传递管道的指针，而不是管道本身。

makechan 接收的 t 类型指针，包含管道中元素的类型信息，接收 size，代表缓冲的容量。

注释已经很清楚，总结：

  • 先根据 size 和元素类型，计算出缓冲区所需分配的内存容量。
  • 如果管道中没有缓冲区，只需为 hchan 结构体本身分配一段内存空间即可。
  • 如果缓冲区的元素不是指针类型，则给 hchan 和 buf 数组分配一段连续的内存空间。
  • 如果缓冲区的元素是指针类型，则 hchan 和 buf 分开分配内存，GC 会扫描管道中的元素。
  • 初始化 hchan 的各个字段，比如元素的大小，元素的类型，缓冲区的容量，最后返回创建好的管道的指针。

向管道发送数据

使用 ch <- x 语句

chansend1 是 c <- x 经过编译后的入口函数。

我们直接看 chansend 就好，看看参数：
c 是你要往里面发送数据的管道，ep 是待发送的数据所在的内存地址，block 表示发送操作是否阻塞式
最后返回的布尔值代表是否发送成功

函数中用的 gopark 函数，做的事是：将当前的 goroutine 置于等待状态（挂起），它让出处理器的使用权，等待被恢复（执行第一个参数 unlockf），如果它传了 nil，那就是永久阻塞了，不会再恢复这个 goroutine。

func gopark(unlockf func(*g, unsafe.Pointer) bool, lock unsafe.Pointer, reason waitReason, traceEv byte, traceskip int) {

可见，向管道发送数据，存在几种情况：

  • 如果管道为 nil
    • 如果发送是阻塞式的，则会一直阻塞这个发送的 goroutine，同时抛错
    • 如果发送是非阻塞式的，阻塞不了，就直接不能发送，返回 false
  • 如果管道不为 nil，且管道是非阻塞的，且管道没有缓冲队列且 recvq 队列为空，或，管道有缓冲队列但已经满了
    • 即没有缓冲区的空间接收数据，也没有接收方接收数据，又不能阻塞，则不能发送，返回 false
  • 如果管道已经关闭，是无法向已关闭的管道发送数据的，直接抛错
  • 如果 recvq 队列非空，说明有接收方在等待，说明此时缓冲队列是空的，直接将发送的数据拷贝给等待中的接收方，绕过缓冲队列
    • 即优先检查 recvq 中是否有等待 “读” 的 goroutine
    • 从 recvq 队列中取出最早处于等待的 sudog，直接将数据拷贝给它
  • 如果 recvq 队列为空，且缓存队列未满，则将数据拷贝到缓存队列的 sendx 索引上
  • 如果 recvq 队列为空，即没有接收方等待接收，且没有缓冲区或缓冲区已满
    • 如果管道是非阻塞式的，直接返回 false，无法发送
    • 如果管道是阻塞式的，那发送操作就阻塞了
      • 获取当前 g 的指针，创建一个 sudog，再挂载这次阻塞发送的有关信息，比如发送的管道，是否在 select 控制体中，待发送的数据的地址等
      • 然后将该 sudog 加入到当前管道的 sendq 队列中，作为阻塞中的发送方
      • 调用 gopark 方法，将当前的 g 阻塞，直到被唤醒
      • g 被唤醒后会执行一些收尾工作，将一些属性重置，并释放 sudog

我们仔细看看直接发送的逻辑，直接把数据发送给等待接收的 goroutine

执行的是 send 函数：

如果接收方 sg 的存放数据元素的地址存在，则直接将 ep 地址上的数据，拷贝 sg.elem

谈谈 goready，它就是唤醒传入的接收方的 goroutine，将它转成 Grunnable 可运行状态，会执行 runqput 函数将当前 runq 队列的末尾，等待被调度，这个 runq 队列存放可运行的 goroutine，

值得注意的是，发送数据只是将接收方的 g 放到了处理器的 runnext 上等待执行，程序没有立即执行该 g

小结一下常见的几种情况：

  • 如果当前管道的 recvq 队列存在阻塞中的接收方，则直接将数据发送给等待中的接收方，将其设置为下一个运行的 goroutine
  • 如果管道的 recvq 队列为空，且存在缓冲区，且还有剩余空间，则直接将数据存到缓冲区的 sendx 所在的位置上
  • 如果不满足上面情况，则会创建一个 sudog，将这次阻塞发送的相关信息挂载到上面，将 sudog 加入到 sendq 队列中，当前 g 陷入了等待，等待其他 g 从管道中接收数据

向管道发送数据，存在几个会触发 g 调度的时机：

  • 发送数据时，如果管道的 recvq 队列中存在等待接收数据的 goroutine，立刻设置处理器的 runnext 属性，但不会立刻触发调度
  • 发送数据时，没有找到接收方，并且缓冲区满了，这时会将自己（goroutine）包裹成 sudog，加入到管道的 sendq 等待队列中，并调用 gopark 触发 Goroutine 的调度，让出处理器的使用权。

从管道接收数据

从管道接收数据有两种方式：
i := <- ch
i, ok := <- ch

经过编译，都会变成 ORECV 类型的节点，但后者会在类型检查阶段被转换成 OAS2RECV 类型。

分别被转换成 chanrecv1 和 chanrecv2 两种函数调用，但是都会实际调用 chanrecv：

chanrecv 函数做的事是：在管道 c 上接收数据，并将接收到的数据写入 ep 指向的内存。

函数的参数：c 是管道，ep 是元素的指针，block 是否阻塞。selected 一般与 select 控制结构配合使用，代表是否进入 case 分支，received 表示已接收到值

从管道接收数据的处理过程：

  • 如果管道为 nil
    • 如果接收是非阻塞的，接收失败，返回两个 false
    • 如果接收是阻塞的，则阻塞这个接收方，而且永久阻塞了
  • 如果管道不为 nil，且管道非阻塞，且管道没有缓冲区且 sendq 队列没有发送方，或管道有缓冲区但缓冲区没有数据
    • 如果管道没有关闭，无法从管道接收数据，返回两个 false
    • 如果管道关闭了，且此时没有可接收的数据，而 ep 可以接收到一个零值，返回 true 和 false
  • 如果管道已经关闭，且管道的缓冲队列中没有数据
    • ep 可以接收到一个零值，返回 true 和 false
  • 如果 sendq 队列非空，获取一个阻塞中的发送方
    • 如果没有缓冲区，则直接将发送方的数据拷贝到 ep
    • 否则将缓冲区中 recvx 索引上的元素拷贝给 ep，再将 sendq 队列的队首的 sudog 的 elem 上的数据拷贝到缓冲区的 recvx 索引上，填补过来
  • 如果 sendq 队列为空，缓冲队列不为空，直接从缓冲队列中拷贝数据给 ep
    • 获取 sendx 位置上的指针，如果 ep 存在，将它拷贝给 ep
    • 将 recvx 上的内存清理，然后更新 recvx 索引，如果正好越界，重新归零，即环形队列
    • 缓冲队列的元素个数 - 1
  • 如果 sendq 队列为空，没有发送方，且缓冲队列为空
    • 如果是非阻塞状态，就无法接收数据，返回两个 false
  • 如果 sendq 队列为空，并且是阻塞的，那么这次接收就是阻塞的
    • 获取当前的 g，创建一个 sudog，将接收数据的 ep 挂载到上面，将 sudog 加入到 recvq 队列
    • 调用 gopark 函数，将当前 g 阻塞，直到被恢复
    • 被唤醒后会进行一些清理工作，释放 sudog，返回 true !closed

仔细看看直接接收
当管道的 sendq 队列中有处于等待的 goroutine 时，会取出队首，调用 recv 函数。

可见

  • 如果管道没有缓冲区，直接调用 recvDirect 函数将 sendq 发送队列中的 sudog 存储的 elem 数据直接拷贝到目标内存地址 ep 中
  • 如果管道有缓冲区，将 sendq 发送队列中的数据拷贝到缓冲区中，释放了一个阻塞的发送方，发送队列的队头 sudog 的 elem 元素，会替换索引 recvx 所在位置的元素，recvx 上原有的元素会被拷贝到接收数据的变量的内存空间中

无论是那种情况，都会调用 goready 函数将当前处理器的 runnext 设置接收方的内存地址

小结一下 从管道接收数据的过程

  1. 如果管道为 nil，直接调用 gopark 挂起当前的 goroutine
  2. 如果管道已经关闭，并且缓冲区没有任何数据，直接返回
  3. 如果管道的 sendq 队列中存在等待的 goroutine，这说明要么不存在缓存队列，要么缓存队列满了，前者直接将等待中的 sudog 中的元素拷贝到接收变量所在的内存空间，后者将数据队列中 recvx 索引上的元素拷贝到目标变量，然后将等待队列 sendq 中的队首的元素，拷贝到 buf 队列的 recvq 索引上、
  4. 如果管道的缓冲区有数据，但未满，说明 sendq 等待队列是空的，直接读取 recvq 索引上的对应的数据，拷贝到 ep，
  5. 。。。

总结一下从管道接收数据时，触发 goroutine 调度的两个时机
当管道为 nil 时
当缓冲区没有数据，并且也不存在数据的发送者时

关闭管道

当你不再使用一个管道（不再往里面发送值）时，别忘了把它关闭。

这就是告知接收方的那个 goroutine，发送工作已经完成。

编译器会将 close(ch) close 这个关键字转成 OCLOSE 节点，接着转成 closechan 函数执行。

所以，管道关闭的核心逻辑

  • 释放所有的接收者，将它们加入 gList 队列
  • 释放所有的发送者，将它们加入 gList 队列
  • 将 glist 中的元素都调用 goready 触发调度

关闭管道后，对于处于等待的接收方来说，会收到一个相应类型的零值
对于等待的发送方来说，会直接 panic

close 的逻辑是先锁住管道，然后把所有的接收方和发送方连成一个 sudog 链表，再解锁
然后再将所有 sudog 逐个全部唤醒，并使其进入 Grunnable 状态
被唤醒的 g，继续执行它的逻辑
接收方会进行一些扫尾工作，返回，selected 返回 true，received 则要根据管道是否关闭，返回不同的值，在这里管道已经关闭，则返回 false
发送方也会继续执行 chansend 函数里 gopark 函数之后的代码，但检测到管道已经关闭，会 panic
